Corten Logistics seeded as tenant zero (`00000000-0000-0000-0000-000000000001`) with all six modules enabled by default so existing functionality keeps working unchanged. Every new freight-engine table from this migration onward is gated by `org_id` + module flag; existing public.* tables get retrofitted in migrations 026-029.
- **`022_rate_engine_core.sql` - Braiin Rates module.** New `rates` schema replacing Wisor / Catapult / Cargocosting at the data layer: `rate_card_header`, `charge_lines`, `rate_breaks`, `feedback_log`, `carrier_profile`, `charge_code_ref`, `composition_queue`. Idempotent composition queue (unique partial index on open jobs prevents duplicate trigger fires), version monotonicity trigger that also enforces org-isolation on supersedes chains, daily expiry cron, three pgmq queues. RLS posture matches 018/020/021. Hot read path indexed on `(org_id, mode, origin_port, dest_port, valid_from, valid_to, review_status, gate_level)`.
- **`023_commercial_intelligence.sql` - Braiin Pulse module.** Creates the `commercial` schema for the moat: `client_profile` (slim - status / account_tier / relationship_owner / onboarded_date stripped, those move to future `crm.companies` in migration 028; identity fields kept as denormalised cache per architectural decision A2), `client_contacts`, `shipments` (`tms_job_ref` relaxed to nullable + non-unique pending Cargowise multi-ref handling in 025), `shipment_charges`, `client_lanes` (with computed `volume_trend` / `gp_trend` from a prior-12-month CTE so volume-drop alerts actually fire), `enrichment_log`, `alerts` (covering dedupe index includes both OPEN and SNOOZED so re-firing on snooze expiry is a single index probe), `account_wiki_pages`. Per-org cron wrappers (`rebuild_all_lanes_for_active_orgs`, `generate_all_alerts_for_active_orgs`) iterate active organisations with per-tenant `EXCEPTION WHEN OTHERS` isolation - one slow or broken tenant cannot block the others. `is_primary_lane` reset is a single atomic UPDATE using a window function rather than two-step (closes the race with the alert cron). `mark_wiki_stale` is statement-level with `REFERENCING NEW TABLE`, so a 10k-row Cargowise sync produces one batched UPDATE per affected client instead of 10k row-level UPDATEs. `ALTER DEFAULT PRIVILEGES IN SCHEMA commercial REVOKE ALL FROM PUBLIC` plus all eight tables and three views in the explicit REVOKE block. Daily expire-old-alerts (was weekly). **Deferred:** the planned cross-module type relocation (move `severity_type`, `confidence_type`, `category_type`, `billable_to_type` from `rates.*` to `core.*`) hit a Supabase event-trigger interference during `CREATE TYPE core.severity_*` that produced an `operator does not exist: core.severity_level = rates.severity_type` error even with renamed types. Pulse therefore continues to reference the four shared types from `rates.*` directly until a follow-up migration unwinds the catalog interference. Per-tenant `pulse_default_alert_owner` config is set during org onboarding, not in this migration.

### Strategy / RFC

- **`docs/rfc/engiine-adoption.md` - draft mapping from external engiine architecture project to Braiin's stack.** The engiine project at `~/ai-projects/engiine/` is a 140-page architecture doc (v0.4) describing a "universal linking layer" with event sourcing, polymorphic nodes, typed directed links, single LLM boundary, and a feedback loop. Spec assumes Python + FastAPI + Docker Compose VPS - wrong stack for Braiin (TypeScript / Next.js / Supabase / Vercel). This RFC translates the load-bearing engiine ideas into Braiin-native artifacts and a 5-week build order: (1) `src/lib/llm-gateway/` as the single LLM boundary with token metering, content-hash cache, multi-provider readiness; (2) decision_id + feedback loop tied to `activity.events`; (3) new `graph` schema with typed directed links + connected-view projections; (4) `config/shorthand.yaml` as canonical freight vocabulary; (5) cheap-path-first lint rule; (6) 19-node-type catalogue as naming north star. Stream Phase 1 (RFC 024, shipped this morning) already covers ~30% of engiine's contract - this RFC formalises the rest. Three checkbox decisions deferred to Rob in section 7. Commits as draft; promote to RFC 026 after sign-off.

### LLM gateway - Phase 1 (engiine borrowing §3.1)

- **`026_llm_gateway.sql` - telemetry + content-hash cache tables in the activity schema.** New `activity.llm_calls` table records every LLM invocation with provider/model/purpose/tokens/cost/latency/cache-hit fields plus soft links for `decision_id` and `correlation_event_id` (for the feedback-loop RFC that follows). Hot read paths indexed: per-purpose monthly cost rollup, failures-in-last-hour, cache hit rate per purpose. New `activity.llm_cache` table is a content-hash response cache keyed by SHA-256 of (provider | model | system | user | params); sits beside Anthropic's 5-minute ephemeral prompt cache rather than replacing it (ephemeral wins inside a 5-minute burst, content-hash wins across hours/days for deterministic prompts). Daily `pg_cron` job at 03:15 UTC drops expired entries. `service_role` grants baked into the migration from day one (lesson from 024/025 silent-failure debacle).
- **`src/lib/llm-gateway/` module - the single boundary every Braiin LLM call goes through.** Public API: `complete({ purpose, system, user, model?, maxTokens?, cacheKey?, cacheTtlSeconds?, requestedBy?, ... })` returns `{ text, inputTokens, outputTokens, cachedInputTokens, model, provider, cacheHit, callId, latencyMs }`. Behaviour: cache-key opt-in lookup against `activity.llm_cache` -> on hit, write a cache_hit telemetry row and return immediately; on miss, call the Anthropic provider, write success or failure row to `activity.llm_calls`, optionally upsert the cache, return result. Cost computation strips Anthropic date-suffixes (`claude-haiku-4-5-20251001` -> `claude-haiku-4-5`) when looking up per-million-token pricing. Module structure: `index.ts` (public API), `providers/anthropic.ts` (the only file in `src/*` that should hit `api.anthropic.com` once all 14 existing call sites are migrated), `cache.ts` (SHA-256 keying, read/write to `activity.llm_cache`, errors logged but never thrown), `metering.ts` (telemetry write, errors logged but never thrown - cache and metering failures must never break the user-facing call), `types.ts` (shared interfaces, `LlmModel = "claude-opus-4-7" | "claude-sonnet-4-6" | "claude-haiku-4-5" | (string & {})` for autocomplete + flexibility on dated snapshots).
- **`src/app/api/classify-email/route.ts` - first consumer migrated to the gateway.** Replaces the direct `fetch("https://api.anthropic.com/v1/messages")` with `llmComplete({ purpose: "classify_email", model: MODEL, system: { text: CLASSIFIER_RULES, cacheControl: "ephemeral" }, user: userMessage, requestedBy: session?.email ?? "service_role" })`. The 5-minute Anthropic ephemeral prompt cache for the static classifier rules block is preserved end-to-end. The error path is now `instanceof LlmGatewayError` -> 502 (unchanged behaviour from the original `if (!res.ok)` 502); other unexpected throws still 500. Net diff: -33 / +21 lines on this route alone, with the boilerplate shifted into the gateway. The other 13 call sites (research, client-report, deal-coach, client-chat, tag-summary, compose-email, braiin-chat, client-reresearch, refine-replies, deal-workspace, classify-batch, enrichment/researcher) follow in subsequent commits; CI lint to block direct `api.anthropic.com` imports outside the gateway is added once the migration sweep is complete.
- **`/dev/llm` telemetry dashboard + `/api/dev/llm-recent` feed.** Read-only operator view of the last 500 rows from `activity.llm_calls` for tenant zero. Three period-summary cards (today / this week / this month) show total cost, call count, cache hit rate, failure count, and tokens in/out. Two breakdown tables aggregate by purpose and by model with calls / cost / avg latency / cache hit %. Recent-calls feed shows the last 100 with cost, latency, and status badges (`ok` / green `cache` / red error code). Auto-refreshes every 10s. Page is `PageGuard`-gated on `dev_llm` (super_admin always allowed). Cost figures come from the per-million-token pricing table baked into the gateway (`PRICE_PER_M_TOKENS_CENTS` in `src/lib/llm-gateway/index.ts`); rough until per-provider invoice reconciliation is wired up but accurate enough to spot anomalies.
- **First sweep batch - 3 of 14 callers migrated to llm-gateway.** `research`, `client-report`, `deal-coach` routes now call `llmComplete()` instead of `fetch("https://api.anthropic.com/v1/messages")`. Each gets a distinct `purpose` tag (`research_analysis`, `client_report`, `deal_coach`) so the dashboard breakdown shows per-feature spend. Error paths preserve the original 502 on provider failure. `ANTHROPIC_KEY` constant removed from each route. Net per route: ~-15 / +10 lines, boilerplate shifted into gateway.

### Stream module - inbound webhook auth path

- **`src/proxy.ts` - allow inbound webhook routes through the session gate.** The global API session middleware was intercepting `/api/inbound/*` before the route handler's `Bearer ${INBOUND_WEBHOOK_SECRET}` check could run, so any CloudMailin POST was rejected with `401 Not authenticated` regardless of the bearer header. Added a one-line bypass mirroring the existing pattern for `/api/auth/*` and `/api/cron/*` - the inbound route enforces its own shared-secret auth in the handler, identical to how cron routes enforce `CRON_SECRET`. Without this fix the entire inbound mail pipeline is dead on arrival; the activity backbone correlation system from migration 024 has nothing to match against.
- **`src/app/api/inbound/email/route.ts` - accept HTTP Basic Auth as well as Bearer.** CloudMailin's Free plan only supports HTTP Basic on the target URL (no custom headers), and Basic is the path of least friction for Resend Inbound / Mailgun Routes / SendGrid Inbound Parse on similar tiers. The handler now accepts either `Authorization: Bearer <INBOUND_WEBHOOK_SECRET>` (existing scheme - any future source that supports custom headers) or `Authorization: Basic <base64("braiin:<INBOUND_WEBHOOK_SECRET>")>` (CloudMailin and the rest). Username is fixed to `braiin` for clarity in CloudMailin's UI; password is the same shared secret. Basic over HTTPS provides equivalent security to Bearer - the choice is purely about which provider can actually send each header.
- **`/dev/activity` smoke-test page + `/api/dev/activity-recent` feed.** Read-only operator view of the last 50 rows from `activity.events` for tenant zero, ordered occurred_at DESC, auto-refreshing every 10s. Rows colour-code by class: blue = outbound, green = matched inbound (correlation hit), orange = orphan inbound (`subject_type='orphan_inbound'`). Three counter cards summarise the breakdown so a quick refresh after a real CloudMailin delivery shows whether the loop populated rows. Page is `PageGuard`-gated on `dev_activity` (super_admin always allowed; explicit `page_access` for others). API is service-role on the activity schema since `activity.*` is locked to service role only. Closes the verification gap the inbound pipeline left behind - we now have a one-click confirmation that the CloudMailin->webhook->backbone chain is intact without poking psql.
- **`025_activity_service_role_grants.sql` - fixes a critical hole left by 024.** Migration 024 correctly revoked all PUBLIC privileges on the activity schema, but omitted explicit grants to `service_role`. Supabase does not auto-grant service_role on user-created schemas - the result was every server-side `.schema("activity").from(...)` call failing at the PostgREST layer with `PGRST106 "Invalid schema: activity"`. Because the inbound webhook's `logEvent()` was wrapped in a `.catch()` that returned 200 regardless, CloudMailin showed "Successful Today: N" while zero rows landed in `activity.events`. The Stream module's entire write path has been silently broken since 024 deployed. This migration grants service_role USAGE on the schema plus SELECT/INSERT/UPDATE/DELETE on tables, EXECUTE on functions, USAGE on types, and the equivalent ALTER DEFAULT PRIVILEGES so future tables added to the schema inherit the access without each migration restating it. **Manual step also required:** add `activity` to Supabase Dashboard -> Settings -> API -> "Schemas that will be exposed via PostgREST" - that is a project-level config a migration cannot set.

### Security hardening (audit follow-up)

#### CRITICAL fixes

- **`/api/classify-batch` now requires manager or super_admin.** The endpoint submits bulk email reclassification jobs to Anthropic - any signed-in viewer was previously able to call it and trigger unbounded API spend against email IDs they did not own. POST and GET are both gated; non-managers receive 403. Tracking rows still record `submitted_by` for audit. `(src/app/api/classify-batch/route.ts)`
- **Tasks table locked to service-role access.** Migration `018_tasks_rls.sql` enables RLS on `tasks`, drops any legacy "Allow all" policies, and revokes table + sequence grants from `anon` and `authenticated`. The `/api/tasks` REST layer (which runs under the server-side service-role key) remains the sole read/write path; the application-level visibility rules already enforced there now have a defence-in-depth layer beneath them. Closes the gap that migration 017 left open.
- **Browser-side direct task insert removed.** `src/hooks/use-email-actions.ts` (the email-pin follow-up flow) now POSTs to `/api/tasks` instead of calling `supabase.from("tasks").insert(...)` from the client. Required because the new RLS policy denies anon writes; also brings the email-pin task creation through the same Zod validation + visibility scope as every other task entry.

#### HIGH severity follow-ups

- **CRON_SECRET hardening (shared helper).** New `src/lib/cron-auth.ts` `requireCronAuth(req)`. An unset / empty `CRON_SECRET` now returns 500 ("deploy misconfiguration") instead of silently disabling the cron with a 403 that looks like normal access denial. Also closes the empty-string bypass where `Bearer ` was a valid token. Both `/api/cron/sync-outlook-tasks` and `/api/cron/poll-classify-batches` route through the helper.
- **`/api/classify-email` bulk hydration path now requires authentication.** Previously the POST `{ bulk: { ids: [...] } }` branch only enforced rate limits, allowing an unauthenticated caller (under the strict 30/min IP bucket) to enumerate email_classifications rows. Returns 401 when no session.
- **`/api/classify-email` PUT writes now have an audit trail.** Migration `019_email_classifications_audit.sql` adds `last_modified_by` + `last_modified_at` to `email_classifications`; the PUT handler stamps both on every write. **This is forensics, not access control:** the original audit's HIGH-severity IDOR (any authenticated user can overwrite any other user's per-row overrides on a shared `email_id` row) is **not** resolved by this change - it remains an **accepted risk** until a per-user override schema is in place. The audit field at least makes the writer attributable. Tracked as a deferred schema redesign.
- **Generic 500 messages instead of leaking `e.message`.** Both classify-email handlers now log full errors server-side and return short generic text to the client. Same change applied to `/api/staff-ai-prefs` and `/api/ai-samples` DELETE.
- **`/api/staff-ai-prefs` scoped by department.** Managers can now only view + modify staff in their own `department`; super_admin remains unrestricted. Mirrors the visibility model used by `/api/tasks`. Cross-department writes return the same 403 shape as the unauthorised case so the API doesn't reveal the org chart.
- **Outlook sync failures surfaced in API responses.** `/api/tasks` PATCH and DELETE now include a `sync_warning` field when the Microsoft Graph round-trip failed, so the UI can show a non-blocking toast instead of silently leaving local + remote out of sync. PATCH still flips `sync_status` to `error` on failure; DELETE proceeds with the local row removal but logs the orphaned remote ID.

#### Type debt cleanup (root-cause fix)

- **`database-extensions.ts` now extends existing tables in addition to declaring missing ones.** A new `Augment<T, Add>` helper composes the per-migration column additions from migrations 011 (relevance dept/mode), 012 (ai_tags / user_tags / relevance_feedback), 013 (conversation_stage), 016 (ai_learning_share_team), 017 (Outlook sync columns), and 019 (audit fields) into the generated row/insert/update shapes. Removed the file-level `as any` cast in `cron/sync-outlook-tasks/route.ts`, the five `as never` casts in `tasks/route.ts`, the `as never` casts in `classify-batch/route.ts` + `cron/poll-classify-batches/route.ts` + `lib/classify-batch.ts`, and the `as any` on the user_preferences query in `classify-email/route.ts`. Local typed shapes replace the `Record<string, unknown>` placeholder in the affected handlers so future changes get real type errors instead of silent drift.

#### Reliability / code quality

- **Shared `determineFinalStatus()` for classify_batches terminal states.** Both `/api/classify-batch` GET and `/api/cron/poll-classify-batches` now call the same helper from `src/lib/classify-batch.ts` so the cron can no longer drift from the manual poll path. The previous implementation also had a logic bug in the all-errored detection (`errored === processing + succeeded`) which never fired - replaced with the obvious check `succeeded === 0 && errored > 0`.
- **`processBatchResults` failures no longer marked as completed.** When the JSONL download or parse fails, we now stamp the tracking row as `errored` with the error message in `notes`, instead of marking it `completed` with `succeeded=0, errored=0` and hiding the failure from the dashboard.
- **Parallel batch polling.** `/api/classify-batch` GET and the cron now poll all open batches with `Promise.all` instead of a sequential `for (const row of rows) await ...` loop, which previously hung the request behind the slowest Anthropic status call.
- **Microsoft Graph app token cached.** `getAppToken()` in `src/lib/outlook-todo.ts` now caches the token in a module-level variable with a 60s refresh buffer, instead of fetching a fresh token from Azure on every Graph operation. A single task create previously triggered 3 token fetches; the 15-min sync cron compounded the cost.

#### Rate limiting + RLS coverage

- **Per-user write rate limits** added to `/api/tasks` (120/min/user covers heavy triage), `/api/classify-batch` (10/min POST, 60/min GET - manager-gated already, this caps frequency), `/api/staff-ai-prefs` PATCH (60/min), and `/api/ai-samples` DELETE (60/min). Uses the existing `checkRateLimit` helper backed by the `check_rate_limit` Postgres RPC so all Vercel workers share one counter.
- **`classify_batches` RLS lockdown.** Migration `020_classify_batches_rls.sql` enables RLS, drops any "Allow all" policies, and revokes anon/authenticated grants on the table + sequence. Same posture as migration 018 on `tasks` - service-role API path becomes the only access route.

#### Re-review follow-ups (caught on the second-pass audit)

- **Anthropic batch `canceling` state guarded.** Both the manual GET in `/api/classify-batch` and the cron poller previously only treated `processing_status === "in_progress"` as transient. A batch in Anthropic's `"canceling"` state would fall through, get written as `"completed"` with zero counts, and never be re-polled (the SELECT filter is `.eq("status", "in_progress")`). Now both paths early-return on either `"in_progress"` or `"canceling"`, so the row stays open until Anthropic transitions it to `"canceled"` with real counts.
- **`Promise.all` -> `Promise.allSettled`** for parallel batch polling. A single unexpected throw from the Supabase client during a per-row update would previously reject the whole tick (and trigger Vercel cron retries on the cron path). Per-batch failures are now logged and surfaced individually without nuking sibling batches.
- **`classify-batch` GET response sanitised.** The `notes` column on `classify_batches` stores raw exception text from `processBatchResults` failures; surfacing that verbatim to the manager UI would leak Anthropic response bodies / network errors. New `sanitiseBatchRow()` helper strips `notes` and replaces it with a boolean `has_processing_error` flag. Applied on every return path of GET.
- **`classify-batch` POST: tighter rate limit + 24h quota.** Per-minute cap dropped from 10 to 3 (a single submission already covers up to 1000 emails). Added a daily quota of 25,000 Anthropic batch requests per `submitted_by`, computed by summing `request_count` over the last 24h. Bounds the daily blast radius of a compromised manager session at a level well above any legitimate backfill.
- **`/api/tasks` PATCH and DELETE manager scope is now department-bound.** Read scope (`?scope=team`) was already department-scoped via the staff table; the write paths called `isManager()` with no department filter, so an Ops manager could PATCH/DELETE a Sales team task. Now both PATCH and DELETE check that the task's assignee or creator is in the manager's department before granting fall-through manager authority. super_admin remains unrestricted.
- **`staff-ai-prefs` PATCH logs misconfig** when a manager has `department = NULL` on their staff row (a likely cause of silent 403s after upgrade). The response stays a flat 403 to preserve the existing access-control shape; the log line lets ops diagnose it from Vercel function logs.
- **Cron-auth 500 response generic.** When `CRON_SECRET` is unset, the response body now says only `Internal server error`. The full diagnosis still goes to server logs, but the env-var name no longer appears in the HTTP response.
- **Stripped two leftover `any` casts** in `src/app/api/classify-email/route.ts` (lines 360, 520) on `feedback` and `senderHistory` map callbacks.

### AI learning governance

- New `/settings/ai-learning` review page (manager + super_admin only) gives visibility and control over the writing-voice corpus that drives reply suggestions. Two sections: per-staff toggle table and a captured-samples browser.
- **Per-staff overrides**: list every active staff member with their current `Capture` and `Share with team` toggles, both editable inline. A manager can disable cross-team sharing for a specific person (e.g. someone leaving, or a junior whose voice we don't want seeding the corpus yet) without waiting for that user to update their own profile. Toggles fire optimistically with rollback on API error.
- **Sample browser**: paginated table of every captured reply with sender attribution (`Adrienne Solyom (Sales)`), date/time, the original email subject and recipient, the reply text (3-line clamp), and an "AI used" badge when the staff member sent the AI's suggestion verbatim. Filter by sender, date window (7 / 30 / 90 / 365 days), and subject search. Per-row delete removes the sample from the corpus so it stops being used as a pattern.
- New endpoints: `GET/DELETE /api/ai-samples` and `GET/PATCH /api/staff-ai-prefs`. All gated to manager / super_admin. Validation via Zod for the PATCH payload so a malformed override can't write garbage to user_preferences.

### Cross-team writing-voice corpus

- Classifier reply suggestions now learn from EVERY Corten staff member's recent replies (not just the current user's), so the AI's drafts match the organisation's house style as the team grows. Each sample is attributed by sender name + department in the prompt (e.g. `RE: Felixstowe quote - Adrienne Solyom (Sales) wrote: "..."`) so Claude can pattern-match by role.
- Privacy guardrails are explicit and per-user. Migration `016_ai_learning_share_team.sql` adds `user_preferences.ai_learning_share_team` (default true) alongside the existing `ai_learning_enabled`. Two toggles in `/profile` -> Preferences -> AI Learning:
  - **Learn from my emails** (existing): when off, your sent replies are not captured into ai_writing_samples at all. Use for personal / sensitive accounts.
  - **Share my replies with the team's AI** (new): when off, your captured replies are still used by your own AI but excluded from other staff's classify-email prompt assembly. Use when you want personal learning without your voice spreading across the team.
- Closes a latent privacy bug. The previous classify-email writingSamples query had no user filter at all - it pulled the most recent 10 samples from anyone, even users who'd opted out of learning. Now the query honours both per-user toggles correctly: a sample is only included if (a) the original sender's `ai_learning_enabled` is true AND (b) `ai_learning_share_team` is true OR the current classify caller is the same staff member.
- Internal-to-internal replies are now excluded from the corpus. A reply sent to another Corten staff member isn't useful pattern data for client-facing suggestions, and including them risked leaking internal tone (slack-style chitchat) into external drafts. Filter happens client-side after the query so the corpus stays focused on outward-facing voice.

The hot-path `/api/classify-email` POST stays sync because it has to be sub-second; this is for backfill / housekeeping where a 5-30 minute turnaround is acceptable (24h SLA).
- Migration `015_classify_batches.sql` tracks batches: `anthropic_batch_id`, `email_ids[]`, status (in_progress / completed / canceled / expired / errored), submitted_by, request_count, succeeded_count, errored_count.
- New `src/lib/classify-prompt.ts` extracts CLASSIFIER_RULES + `buildBatchUserMessage` so sync and batch share the same prompt and don't drift apart over time.
- New `src/lib/classify-batch.ts` wraps Anthropic's Batches API: `submitClassifyBatch(emails)` posts the batch with cached system prompt; `fetchBatchStatus(id)` polls; `processBatchResults(id, url)` downloads JSONL output, parses each line's classification, normalises tags + stage, and upserts `email_classifications`.
- New `/api/classify-batch` route: `POST {email_ids[]}` submits, `GET ?id=N` polls a single batch, `GET ?all=open` polls every in-flight batch (used by the cron + a manual button). All authenticated.
- New cron `/api/cron/poll-classify-batches` registered every 5 minutes in `vercel.json`. Iterates open batches, processes any that have ended, marks tracking rows complete with success/error counts. Secured by `CRON_SECRET`.
- New `Bulk classify (Anthropic Batch API)` panel at the top of `/usage`: live count of stale rows (NULL ai_tags or NULL ai_conversation_stage), a `Reclassify N stale rows` button that batches up to 1000 ids in one submission, a `Poll now` button for impatient checks, and a list of recent batches with status, counts, and submitted_at.

### Security

- JWT session signing: the `braiin_session` cookie is now a signed HS256 JWT instead of raw JSON. Forged cookies are rejected cryptographically at the proxy (edge) layer. Closes auth bypass where any non-empty cookie passed the previous presence check.
- Azure `access_token` and `refresh_token` removed from the JWT payload and the `Session` type. They were never read anywhere (all Graph calls use app-level `client_credentials`), so keeping them in the cookie only served as a leak vector for Mail.Read/Mail.Send scope tokens. If user-delegated Graph access is ever needed in future, store tokens in a dedicated encrypted table keyed by staff_id - not in the session cookie.
- `SESSION_SECRET` is now required at startup and must be at least 32 characters. The previous fallbacks (`SUPABASE_SERVICE_KEY`, literal `"fallback-dev-secret-change-me"`) have been removed - the app now refuses to start without a valid secret.
- Renamed `src/middleware.ts` to `src/proxy.ts` (Next 16 convention). The proxy cryptographically verifies the JWT via `verifySessionToken` on every `/api/*` request except `/api/auth/*` and `/api/cron/*`. Expired tokens return 401.
- Added shared `getSession()` helper in `src/lib/session.ts` that verifies the JWT and returns a typed `SessionPayload`. Replaced 7 duplicated route-local `getSession()` functions that did unsafe `JSON.parse` on the raw cookie value.
- `auth/session` endpoint now logs Supabase errors when looking up staff access rather than swallowing them silently.
- `send-email` now reads sender from the verified session, never from the request body. Closes the sender-spoofing vector where any authenticated user could send email appearing to come from any address. Also requires the session email to be an internal domain (403 otherwise), returns 401 if unauthenticated, and no longer leaks the full Resend error payload in the response.
- `email-composer` client no longer sends `from_email` / `from_name` (they were being ignored server-side after the fix).
- Consolidated Supabase client: 21 API routes now import `supabase` from `@/services/base` instead of each instantiating their own client. Closes the silent privilege-downgrade where `SUPABASE_SERVICE_KEY || NEXT_PUBLIC_SUPABASE_ANON_KEY` would fall through to anon if the service key was missing.
- `services/base.ts` now throws at startup if `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` are missing, or if `SUPABASE_SERVICE_KEY` is missing on the server. No more silent fallbacks.
- `src/lib/supabase.ts` reduced to a thin re-export of `@/services/base`'s supabase for backwards compatibility with existing client imports.
