# Changelog

All notable changes to the Braiin dashboard.

## [Unreleased]

### Quoting engine v2 - RFQ inbox mock-up

- **`/dev/quote-inbox` - visual mock-up of the RFQ triage inbox.** Static page, no backend calls. Sits ABOVE the per-quote workspace at `/dev/quote-preview` - the air-traffic-control view across all open RFQs, not the deep-work view. Single sortable / filterable table (denser than Kanban, handles 50 rows): Customer / Lane / Mode / Status / Source / Time-in-state / Detail / Action. Rows colour-tagged by `DraftStatus` (new / gathering / ready / sourcing / recommended / sent / won / lost / expired). Stale highlighting: rows in `ready` more than 30 minutes or `gathering` more than 60 minutes turn rose. KPI strip at the top counts open RFQs / awaiting customer / sourcing (with pulsing brain) / ready to send / stale > SLA. Group filter pills (Open / Awaiting / Closed / All) with live counts. Search box filters by customer, lane, mode, ID. Auto-refresh "live" indicator placeholder.

- **Slide-out RFQ composer (the killer UX move).** Triggered by *Send RFQ* on a `ready` row. Right-hand panel slides in over a dimmed backdrop without navigation - dashboard stays mounted underneath. AI-suggested carriers pre-checked (top 8 by composite score for this lane), each shown with score, last response timing on this lane, response rate %, and source badge (API / Email / Aggregator). Customer-nominated carriers tagged. Three switches: toggle individual carriers, "Customer nominated only" shortcut, "Show all carriers" expander. Auto-generated email template per carrier rendered in monospace preview. Footer shows estimated response count within 2h plus a single Send-to-N-carriers action button.

- **Slide-out "Ask for more info" panel.** Triggered on `new` / `gathering` rows. Lists missing fields as checkboxes, generates a reply draft auto-addressed to the customer's first name with bullet list of the missing items, sends on the existing email thread so the reply auto-classifies and updates the same draft. Decision-loop note explains the operator's edits feed back into the prompt.

- **Sibling-group rendering for multi-quote-per-email.** A single inbound RFQ can spawn N drafts when the customer asks for multiple options upfront (e.g. air Express / Standard / Economy / Charter / Hand-carry on the same lane, or "air OR LCL" alternatives). The inbox renders these as a single parent row with a chevron, an `N options` violet badge, the AI split-confidence percentage, and a per-status summary ("3 sourcing · 1 gathering · 1 recommended"). Children expand inline below the parent with a violet left-border, indented with `↳`, sharing the lane / customer / source columns. Parent's time-in-state mirrors the OLDEST sibling so the group never disappears off the operator's radar. Each child carries its own status, its own detail (sourcing progress / recommendation / missing fields), and its own action button - so the operator can `Send RFQ` on the Express option while still `gathering` on Charter without losing the group context. Demo entry: Apex Pharma DEFRA-USORD with 5 air-product options across 4 statuses.

- **`needs_input` status** added to the state machine. Distinct from `gathering` (waiting on customer for missing fields) - this is `waiting on operator action` for things the AI cannot resolve automatically: a delivery / haulage rate that has to be sourced manually, a spot rate from a niche carrier outside the standard fan-out, an internal SME question (e.g. ops manager confirming HS classification), or a customer question that needs a human answer. Reachable from any active state (`new` / `gathering` / `ready` / `sourcing` / `recommended`). When the input arrives, the draft returns to whichever state it came from. Inbox UX: orange status pill, dedicated `Needs your input` KPI tile that turns orange when > 0, dedicated filter pill, stale-row threshold of 30 minutes (it's blocking work), action button is `Provide input` instead of the default. Detail column lists each pending input by kind (Delivery rate / Spot rate / Question / Haulage / Other) and assignee. Demo entry: Lansdowne Manufacturing CNSHA-GBLEE FCL with two open inputs - haulage rate to Leeds (Marcus) and HS classification question (Sarah).

- **Slide-out `Provide input` panel.** Third operator-action slide-out, completing the visual contract for the inbox. Opens from a `needs_input` row's action button. Header echoes draft context (customer, lane, mode, equipment). Orange banner reminds the operator how many inputs are blocking the draft and that the draft auto-flips back to its previous state when all are resolved. Each pending input renders as its own card with a kind-coloured left border (amber for haulage / delivery rate, cyan for spot rate, violet for question, zinc for other), kind-specific answer fields:
  - **delivery_rate / haulage / spot_rate**: structured fields - currency dropdown (GBP / EUR / USD), amount input, carrier / supplier name, valid-until date.
  - **question**: 4-row free-form textarea.
  - **other**: 3-row notes textarea.
  
  Per-card actions: `Reassign`, `No longer needed` (cancel), `Mark answered`. Resolved cards collapse to a one-line summary with a `Reopen` link. Footer: live count of answered vs open, `Save & close` (preserves state, leaves draft in `needs_input`), and `Resolve & unblock` (disabled until ALL inputs are answered or cancelled, turns emerald when ready).

- **Low-confidence sibling-split review screen at `/dev/quote-split-review`.** Safety valve for the multi-quote-per-email flow. When `classify-email` proposes a split with `split_confidence < 0.85`, the children stay in pending review and the inbox parent row swaps the `Expand` action for an orange `Review split` CTA pointing at this page. Layout: original email on the left rendered sentence-by-sentence with phrase highlights showing which sentences fed which proposed sibling (hovering a sibling on the right dims unrelated sentences and rings the matching ones in amber). Customer-context card below explains why confidence was lowered (e.g. "this customer's last 8 quotes were combined door-to-door, but this email says 'separately'"). Right column: N proposed sibling cards with intent label, mode badge, lane, equipment / weight / commodity / incoterms / collection date, per-sibling confidence pill, and the trigger sentences quoted in italics. Per-card actions: `Edit fields`, `Merge with another`, `Drop this option`. Top-level actions in the sticky footer bar: `Cancel`, `Merge into one quote` (collapses children, treats as single quote), `Confirm split & create N drafts`. Demo: Bewlay Industrial ITGOA-GBLEE - sea leg + road delivery, 62% AI confidence, customer pattern is normally combined door-to-door so operator review needed. Resolution screen confirms what will happen and offers a re-run for the demo.

- **Inbox surfaces low-confidence groups distinctly.** Parent rows where `splitConfidence < 0.85 && !operatorReviewed` show:
  - Orange "needs review" badge next to the AI confidence percentage.
  - Detail column reads "AI suggested split - confirm before drafts go live" instead of the per-status summary.
  - Right-most action is a solid orange `Review split` button linking to `/dev/quote-split-review` (instead of `Expand`).
  
  Once `operatorReviewed` is set, the row reverts to the standard expandable group layout. Demo entry: Bewlay Industrial Genoa-Leeds with two children (Sea Genoa-Felixstowe, Road Felixstowe-Leeds), 62% confidence.

- **Inbox column reorder - action first, status second, source demoted.** The action button used to live on the far right of the row and was easy to lose at scale. Reordered: **Action** (180px, leftmost, full-width solid CTA per status), **Status** (120px, the colour-tagged pill), **Customer & lane** (flex, customer name + lane in mono + ID + YTD on three rows), **Mode** (140px), **Time** (110px, with received-time below in muted tone), **Detail** (260px, sourcing progress / recommendation / missing fields / pending inputs), **Source** (40px, icon-only on the far right with tooltip). Buttons bumped from `h-7` to `h-8` and given `justify-start` + per-status hue (sky for Send RFQ, orange for Provide input, emerald for Review & send, indigo for Mark won, amber for Ask for info, violet for View grid / Expand). The operator's eye now lands on the action immediately, with status as the supporting "why" right next to it. Same pattern applies to single rows, parent group rows (where the action is `Expand (N)` or `Review split` for low-confidence groups), and child rows (where the action is the per-status CTA, indented by 16px).

- **Inbox combines action + status, shrinks pills.** The action button's hue + label already encodes the status (sky = ready, orange = needs_input, emerald = recommended, etc.) so the separate Status column was just visual noise next to it. Dropped the Status column entirely. For terminal states (won / lost / expired) where there is no action for the operator, the action cell renders a small status pill instead so the column always answers "what's this row's state?". Group parent rows fold the `N options` and `needs review` indicators into the customer cell as small inline badges. Introduced a `PILL_SM` class (10px text, 18px height, tighter padding, normal weight) used everywhere a Badge appears so pills no longer compete with the action button for attention.

- **`PILL_SM` applied to per-quote workspace + split review.** Same 18px / 10px pill sizing now used across `/dev/quote-preview` and `/dev/quote-split-review`. ScoreBadge, StatusPill (received / timeout) and SourceBadge in the live RFQ grid all shrunk. Top-bar "sourcing" status, the BR-… reference (now a Badge instead of a custom span), and the recommendation `pick` badge all on the new size. Confidence percentages and the `GRP-…` reference on the split-review page match. The visual contract is now consistent: bold = action, small grey = label.

### Quoting engine v2 - Phase A: partners + scorecards

- **Migration `038_partners_scorecards.sql` - the carrier rolodex + 5-axis scorecard system.** Five tables. `partners.carriers` is the rolodex itself: name, kind (`carrier` / `agent` / `broker` / `nvocc` / `aggregator`), SCAC + IATA + ICAO codes, transport_modes array (drives selection - sea-only carriers never appear for an air RFQ), `contracting_method` (`api` / `email` / `aggregator` / `portal`), `is_manual` flag for inline-added entries, GIN index on transport_modes, partial indexes on SCAC/IATA/manual. `partners.carrier_contacts` is one-to-many for emails / phones / portal URLs / API endpoints with method + purpose + priority. `partners.scorecards` is per (org, carrier, mode) with five axes 0-100 (Suitability / Speed / Accuracy / Price / Service), composite_score, confidence, plus 90-day sample-size counters (rfq_count / reply_count / job_count / incident_count) and median_reply_minutes. NULL axes treated as 50 (neutral) so brand-new carriers can still surface. `partners.scorecard_weights` lets each org configure the composite weights; default is 0.20 each, CHECK enforces sum = 1.0. `partners.lane_stats` is per (carrier, origin_country, destination_country, mode) with rfq / reply / win counters, avg_rate_rank, on_time_pct - drives the suitability axis for a SPECIFIC lane. `quotes.draft_carrier_selections` is the join table on the quotes side capturing which carriers were sent an RFQ for which draft, with `provenance` tracking (`ai_suggested` / `operator_added` / `operator_kept`) and `overridden_by_operator` boolean. The `partners.suggest_carriers(p_org_id, p_mode, p_origin_country, p_dest_country, p_top_n, p_min_score)` function is the AI's selection oracle: returns ranked carriers for a lane with composite + per-axis scores, lane-history flag (carriers we've used on THIS lane jump up), and reply-rate metadata. Idempotent guards. Manual: apply 038 in Supabase SQL editor and add `partners` to Settings > API > Exposed schemas.

- **Send RFQ slide-out distinguishes AI-suggested vs manually-added carriers.** Two visually distinct sections in the panel. The top section shows AI-suggested carriers (top 8 by composite score on this lane) with their per-carrier score, last reply time, response rate, and source badge - same as before but headed `AI suggested (N of M selected)` with a sparkles icon. A new section below `Manually added (N)` lists operator-added entries on emerald-tinted rows with a `manual` badge and an optional `save to rolodex` badge. An emerald-bordered `Add carrier not on the list` button opens an inline form with name + quotes-email fields and a `Save to rolodex` checkbox - submitting adds the carrier to the RFQ and (if the toggle is on) writes a row to `partners.carriers` with `is_manual=true`. Footer split: `N total · M AI + N manual` so the operator sees the breakdown at a glance, plus the existing "expected replies within 2h" hint. Same flow will land in production wired to `partners.suggest_carriers()` for the AI list and `INSERT INTO partners.carriers ... RETURNING` for the manual-add path.

- **`/dev/carriers` rolodex page.** Static mock of the carrier directory the AI selects from. KPI strip (Total / Manually added / Aggregators / API integrations) above a sortable / filterable table: Carrier (with kind + manual / SCAC / IATA / lane count / last-used), Modes (mode icons - ship / plane / truck), Composite score, the five axis scores (Suit / Speed / Acc / Price / Svc) each as a small score pill colour-coded by tier (≥80 emerald, ≥70 amber, ≥60 orange, otherwise zinc), 90d activity (replies/RFQs and median reply time), and contracting method (api / email / aggregator / portal pill). Search across name / SCAC / IATA / notes. Mode + kind filters. Sample data covers 17 carriers - 6 ocean, 5 air, 2 aggregators, 3 road, plus 2 manually-added entries (Northern Pallet Leeds added by Marcus, Bluewater Air Cargo added by Rob). Footer card explains the composite formula and that manual additions start at 50/50/50/50/50 and gather data from their first RFQ.

### Quoting engine v2 - margin types, multi-currency, validity

- **Migration `039_fx_rates_quote_output.sql` - the FX + multi-currency + validity foundation.** Three layered additions. **(1)** `geo.fx_rates` is a daily FX rate store keyed by `(base_currency, quote_currency, rate_date, source)` with `source_priority` so we can prefer XE.com when available and fall back to exchangerate.host / open.er-api.com when not. `geo.convert_amount(amount, from, to, on_day)` does the lookup with a 7-day sliding window (covers a missed daily fetch), inverse-rate fallback (asked USD->GBP, only have GBP->USD), and triangulation via USD when no direct pair is on file. Returns NULL when no rate is available within 7 days so the operator sees "FX unavailable" rather than a stale silently-used rate. **(2)** `quotes.drafts` gains `quote_output_currency` (GBP / USD / EUR / AUD - default org home), `quote_validity_days` (CHECK in 7 / 14 / 21 / 30), and `quote_valid_until` (explicit date override) with a trigger that keeps days + valid_until coherent so a UI mistake never produces an inconsistent draft. Also adds `container_count` and `pallet_count` quantity columns referenced by per-unit margins. **(3)** `quotes.charge_lines` is the per-spot-response charge table introduced in this migration with the seven-method margin model: `pct` (% of cost), `flat` (fixed amount per line), `per_cbm` / `per_kg` / `per_container` / `per_pallet` (multiplier × draft quantity), and `override` (sell set directly via `sell_amount_override`). `cost_amount` (operator-editable, marked via `cost_overridden_by_operator` boolean), `cost_amount_extracted` (original from extractor, never overwritten - audit trail). `cost_currency` and `margin_currency` per line so a quote can mix USD ocean freight + GBP UK local + EUR origin charges and still convert cleanly to the chosen output currency. Manual: apply 039 in Supabase SQL editor.

- **Per-line margin types in the breakdown panel: `% / flat / per CBM / per kg / per container / per pallet / override`.** The single % input is now a margin-type dropdown plus a value input that adapts. For per-unit types the panel multiplies by the draft's quantity (volume_cbm / weight_kg / container_count / pallet_count) and shows `× N` next to the value so the operator sees the math (`£15 / CBM × 50 = £750 added`). Flat and per-unit margins each have their own currency selector (so an operator can add a flat £20 GBP margin on top of a USD-quoted line - panel converts before summing). `Override` mode lets the operator type a sell amount directly; the panel pre-populates with the current computed sell so toggling between margin types doesn't shock the customer total.

- **Cost AND sell are both editable.** 90% of cost lines come from the carrier extractor and stay untouched, but typo correction or manual rate entry needs to be one keystroke away. Cost field has its own currency selector + amount input; editing sets `cost_overridden_by_operator=true` and the line gets a small `cost edited` amber pill. Sell field is in emerald (the standout colour) and editable directly; typing a sell value switches `margin_type` to `override` and stores the amount in `sell_amount_override` so the schema records the operator's intent cleanly. A `sell set` amber pill appears on lines where the sell was set directly. Rate currency, margin type, value, and currency all visible in one row so the operator can see and tweak everything without expanding anything.

- **Output currency chooser at panel top - `GBP / USD / EUR / AUD`.** Four pill buttons in the top bar pick which currency the customer-facing total is denominated in. Conversions use the day's mid-market rate from `geo.fx_rates` (faked in the mock but identical interface) with the source + timestamp shown beneath the chooser ("FX from XE · 2026-04-28 09:14 UTC"). Cost / Margin / Sell totals at the top all re-render in the chosen currency. Lines whose cost currency differs from the output currency show a `≈ £222` preview next to the editable sell so the operator sees both the carrier-side sell AND the converted figure. Customer view header reads `GBP amount` etc and the footnote explains "charges originally invoiced in their respective currencies, converted at today's mid-market rate".

- **Quote validity chooser - `7 / 14 / 21 / 30 days` or specific date.** Four pill buttons + an inline date picker. Picking a pill sets `quote_validity_days`, picking a date sets `quote_valid_until` directly (and the trigger blanks the other field so they stay consistent). Customer view shows the computed valid-until date in the footer of the quote.

- **Realistic mixed-currency mock data.** The Hapag-Lloyd Standard breakdown now mixes USD (ocean freight $1,660, BAF $226, surcharges, destination charges) with GBP (UK origin THC, doc fee, customs clearance) so the multi-currency conversion is visibly meaningful. Per-line examples now showcase three margin types in the default state: `pct` (most lines), `flat` (£25 on the GBP doc fee, £10 on customs clearance), `per_container` (£50 × 2 containers on the haulage line).

- **Tightened breakdown panel header to one line.** The previous header used pill grids that wrapped awkwardly inside narrow column allocations (currency symbols and codes stacking on separate lines, "7 days" splitting across two lines). Replaced with compact native dropdowns that handle the four currency / four validity options without wrapping: `Quote in [GBP £ ▾]` and `Valid for [14 days ▾] until 2026-05-12`. Cost / Margin / Sell totals now sit inline on the same row, separated by middle dots, so the entire control + summary strip is one row tall instead of three. FX timestamp drops to a single muted footer line beneath. Custom date option appears as a date input only when the operator picks `Custom date` from the validity dropdown, instead of always taking up space.

- **Macro-group + per-currency breakdown structure.** Charges now group into four macro-groups in customer-facing order: **Origin and EXW** (origin / pickup / export customs), **Freight** (freight / mandatory carrier surcharges), **Destination and Delivery** (destination / delivery / import customs), and **Insurance and Other**. Each macro-group has its own colour-tagged left border (amber / violet / cyan / zinc) and shows per-currency subtotals in their native currency on the operator side (e.g. Origin and EXW: £280 GBP + $45 USD ≈ £316). Below the line items, a `Charges by currency` summary shows the total per currency (`GBP £282`, `USD $2,830`) followed by the all-in total in the chosen output currency. **Customer view options panel** at the bottom of the operator side has four toggles: show macro-group sections, show per-currency subtotals inside each section, show charges-by-currency summary at the end, and the always-on "show all-in total in chosen currency". Defaults: macro sections ON, per-currency subtotals OFF (extra detail, opt-in), charges-by-currency summary ON, all-in always shown. Customer view re-renders live as toggles flip - operator sees exactly what the customer will see in the quote PDF / email cover.

- **Indicative charges + caveats - charges that ride the quote without being in the total.** New schema columns on `quotes.charge_lines`: `is_indicative BOOLEAN` and `caveat_note TEXT`. Indicative lines are shown to the customer for awareness (demurrage if held over X free days, detention if container not unloaded promptly, customs duty estimate) but do NOT contribute to per-currency totals or the all-in figure. Operator side: a sky-blue `Info` toggle button per line flips the line indicative; a sky-tinted text input for the customer-facing caveat appears beneath the line when indicative is on. An `indicative` pill appears on the line. Excluded from the macro-group native-currency subtotals and from category-level cost subtotals automatically. Customer view: dedicated `Indicative charges (not included in total)` section between the totals body and the all-in total, with each line's caveat note in italic muted text. New display toggle "Show indicative charges section" in the Customer view options panel (default ON). Demo entries seeded: destination demurrage at $200/container/day after 7 free days at Shanghai port; container detention at $80/container/day after 4 free days at consignee; import customs duty estimated 5-12% of CIF payable to Chinese customs at clearance. Partial index `idx_quotes_charge_lines_indicative` added in migration 039.

- **Roll-up "includes" caption on consolidated rows.** When multiple lines roll up under a shared group label (e.g. THC + DOC + Examination + AMS = "Origin charges"), the customer view now shows a small italic muted caption beneath the consolidated row listing what's inside: `includes Terminal handling - Felixstowe, Documentation fee, Container examination charge (UK), AMS / ENS filing fee`. Maintains transparency without forcing the customer to see every individual line. Same behaviour applies in both the macro-grouped and flat (toggle off) customer view modes.

### Quoting engine v2 - charge code dictionary, multi-TMS mapping, margin rule engine

- **Migration `040_charge_codes_margin_rules.sql` - five things in one migration.** **(1)** `quotes.charge_codes` is the canonical Braiin charge dictionary - one row per Braiin-side charge with `braiin_code` PK, `description`, `billing_type` (`margin` / `revenue` / `disbursement`), `macro_group`, `default_margin_pct`, `applicable_modes` and `applicable_directions` arrays, and `tms_origin` so we know which TMS this code was lifted from. **(2)** `tms.charge_code_map` is the per-TMS translation layer (same pattern as `tms.identity_map` from migration 035) - one row per (provider_id, tms_code) -> braiin_code mapping with `tms_metadata` JSONB for provider-specific attributes (e.g. CW department filter). New TMS adapters (Magaya, Descartes, Magaya etc) just add rows with their own `provider_id`. **(3)** `quotes.margin_rules` is the rule engine: per-org rules scoped by customer / carrier / mode / direction / origin_country / destination_country / origin_unlocode / destination_unlocode / macro_group / charge_code with `rule_priority` auto-computed as a `GENERATED ALWAYS AS (count of non-NULL scope fields) STORED` column so most-specific wins without explicit ordering. Seventeen markup methods covering Wisor + Cargowise: `pct`, `flat`, `per_cbm`, `per_kg`, `per_chargeable_weight`, `per_wm`, `per_container`, `per_container_20`, `per_container_40`, `per_pallet`, `per_bill`, `per_hs_code`, `per_shipment`, `pct_of_line`, `currency_conditional` (with `currency_rates` JSONB - "+10 GBP / +15 USD / +15 EUR"), `override`, `on_cost`. Plus `min_charge_amount` floor, `max_sell_amount` ceiling, `min_margin_pct` floor, and `apply_above_annual_gbp` volume gating. **(4)** `quotes.charge_lines` extended with `billing_type`, `charge_code` FK to canonical, `rule_id` FK to margin_rules (so any sell amount on the customer view can answer "why this margin?" by jumping to the rule that produced it), and `min_charge_amount`/`min_charge_currency` per line. The `margin_type` CHECK constraint is dropped and re-added with the wider 17-method set. **(5)** `quotes.drafts.direction` enum (`import` / `export` / `crosstrade`) so margin rules can scope by direction. Manual: apply 040 in Supabase SQL editor.

- **Seed: 107 canonical Braiin charge codes from the Cargowise dictionary.** Auto-generated from `docs/wisor/Charge codes_CW(1)_UPDATED.xlsx` (CW's full charge code list with billing type and department filter). Each CW code mapped to a snake_case Braiin canonical code (`AFRT` -> `air_freight`, `DTHC` -> `destination_terminal_handling_charges`, `BAF` -> `bunker_adjustment_factor`), with billing_type translated from CW's `MRG` / `REV` / `DSB` to Braiin's `margin` / `revenue` / `disbursement`, macro_group curated per code, and `applicable_modes` / `applicable_directions` derived from CW's department filter (`FEA` -> air export, `FIS` -> sea_fcl import, etc.). All 107 simultaneously seeded into `quotes.charge_codes` AND `tms.charge_code_map` with `provider_id='cargowise'` so the translation layer is live from day one.

- **`/dev/charge-codes` page - the canonical dictionary surface.** KPI strip shows total / margin / revenue / disbursement counts. Search across braiin code, description, and CW code. Filter by billing type, macro-group section, and mode. Table shows: Braiin code (mono), description, billing-type pill (emerald margin / violet revenue / amber disbursement), section pill, mode icons (ship / plane / truck), direction tags, default markup %, and the CW mapping inline (`CW AFRT (FEA,FIA)`). Bottom three cards explain each billing type. Production reads `quotes.charge_codes` JOIN `tms.charge_code_map`.

- **`/dev/margins` page - the margin rule engine surface.** Two views in one page: **Matrix view** is the 3×3 (mode × direction) grid mirroring the Wisor Profit Guide layout, with each cell showing the rules that apply to that mode-direction combination, colour-tagged by macro-group, with the markup method shown as a mono pill (e.g. `+£50/cont`, `+15%`, `min £35`). **Rules list view** is a sortable / searchable list of all rules with their `rule_priority` count badge (number of non-NULL scope fields), scope tags (customer / carrier / mode / direction / macro / charge code), formatted markup, and guardrail (min charge if set). Bottom legend explains all 17 markup methods. ### Stream module - inbound webhook auth path

- **`src/proxy.ts` - allow inbound webhook routes through the session gate.** The global API session middleware was intercepting `/api/inbound/*` before the route handler's `Bearer ${INBOUND_WEBHOOK_SECRET}` check could run, so any CloudMailin POST was rejected with `401 Not authenticated` regardless of the bearer header. Added a one-line bypass mirroring the existing pattern for `/api/auth/*` and `/api/cron/*` - the inbound route enforces its own shared-secret auth in the handler, identical to how cron routes enforce `CRON_SECRET`. Without this fix the entire inbound mail pipeline is dead on arrival; the activity backbone correlation system from migration 024 has nothing to match against.
- **`src/app/api/inbound/email/route.ts` - accept HTTP Basic Auth as well as Bearer.** CloudMailin's Free plan only supports HTTP Basic on the target URL (no custom headers), and Basic is the path of least friction for Resend Inbound / Mailgun Routes / SendGrid Inbound Parse on similar tiers. The handler now accepts either `Authorization: Bearer <INBOUND_WEBHOOK_SECRET>` (existing scheme - any future source that supports custom headers) or `Authorization: Basic <base64("braiin:<INBOUND_WEBHOOK_SECRET>")>` (CloudMailin and the rest). Username is fixed to `braiin` for clarity in CloudMailin's UI; password is the same shared secret. Basic over HTTPS provides equivalent security to Bearer - the choice is purely about which provider can actually send each header.
- **`/dev/activity` smoke-test page + `/api/dev/activity-recent` feed.** Read-only operator view of the last 50 rows from `activity.events` for tenant zero, ordered occurred_at DESC, auto-refreshing every 10s. Rows colour-code by class: blue = outbound, green = matched inbound (correlation hit), orange = orphan inbound (`subject_type='orphan_inbound'`). Three counter cards summarise the breakdown so a quick refresh after a real CloudMailin delivery shows whether the loop populated rows. Page is `PageGuard`-gated on `dev_activity` (super_admin always allowed; explicit `page_access` for others). API is service-role on the activity schema since `activity.*` is locked to service role only. Closes the verification gap the inbound pipeline left behind - we now have a one-click confirmation that the CloudMailin->webhook->backbone chain is intact without poking psql.
- **`025_activity_service_role_grants.sql` - fixes a critical hole left by 024.** Migration 024 correctly revoked all PUBLIC privileges on the activity schema, but omitted explicit grants to `service_role`. Supabase does not auto-grant service_role on user-created schemas - the result was every server-side `.schema("activity").from(...)` call failing at the PostgREST layer with `PGRST106 "Invalid schema: activity"`. Because the inbound webhook's `logEvent()` was wrapped in a `.catch()` that returned 200 regardless, CloudMailin showed "Successful Today: N" while zero rows landed in `activity.events`. The Stream module's entire write path has been silently broken since 024 deployed. This migration grants service_role USAGE on the schema plus SELECT/INSERT/UPDATE/DELETE on tables, EXECUTE on functions, USAGE on types, and the equivalent ALTER DEFAULT PRIVILEGES so future tables added to the schema inherit the access without each migration restating it. **Manual step also required:** add `activity` to Supabase Dashboard -> Settings -> API -> "Schemas that will be exposed via PostgREST" - that is a project-level config a migration cannot set.

#### CRITICAL fixes

- **`/api/classify-batch` now requires manager or super_admin.** The endpoint submits bulk email reclassification jobs to Anthropic - any signed-in viewer was previously able to call it and trigger unbounded API spend against email IDs they did not own. POST and GET are both gated; non-managers receive 403. Tracking rows still record `submitted_by` for audit. `(src/app/api/classify-batch/route.ts)`
- **Tasks table locked to service-role access.** Migration `018_tasks_rls.sql` enables RLS on `tasks`, drops any legacy "Allow all" policies, and revokes table + sequence grants from `anon` and `authenticated`. The `/api/tasks` REST layer (which runs under the server-side service-role key) remains the sole read/write path; the application-level visibility rules already enforced there now have a defence-in-depth layer beneath them. Closes the gap that migration 017 left open.
- **Browser-side direct task insert removed.** `src/hooks/use-email-actions.ts` (the email-pin follow-up flow) now POSTs to `/api/tasks` instead of calling `supabase.from("tasks").insert(...)` from the client. - **CRON_SECRET hardening (shared helper).** New `src/lib/cron-auth.ts` `requireCronAuth(req)`. An unset / empty `CRON_SECRET` now returns 500 ("deploy misconfiguration") instead of silently disabling the cron with a 403 that looks like normal access denial. Also closes the empty-string bypass where `Bearer ` was a valid token. Both `/api/cron/sync-outlook-tasks` and `/api/cron/poll-classify-batches` route through the helper.

- **Shared `determineFinalStatus()` for classify_batches terminal states.** Both `/api/classify-batch` GET and `/api/cron/poll-classify-batches` now call the same helper from `src/lib/classify-batch.ts` so the cron can no longer drift from the manual poll path. - **Per-user write rate limits** added to `/api/tasks` (120/min/user covers heavy triage), `/api/classify-batch` (10/min POST, 60/min GET - manager-gated already, this caps frequency), `/api/staff-ai-prefs` PATCH (60/min), and `/api/ai-samples` DELETE (60/min). #### Re-review follow-ups (caught on the second-pass audit)

- **Anthropic batch `canceling` state guarded.** Both the manual GET in `/api/classify-batch` and the cron poller previously only treated `processing_status === "in_progress"` as transient. A batch in Anthropic's `"canceling"` state would fall through, get written as `"completed"` with zero counts, and never be re-polled (the SELECT filter is `.eq("status", "in_progress")`). Now both paths early-return on either `"in_progress"` or `"canceling"`, so the row stays open until Anthropic transitions it to `"canceled"` with real counts.
- **`Promise.all` -> `Promise.allSettled`** for parallel batch polling. A single unexpected throw from the Supabase client during a per-row update would previously reject the whole tick (and trigger Vercel cron retries on the cron path). Per-batch failures are now logged and surfaced individually without nuking sibling batches.
- **`classify-batch` GET response sanitised.** The `notes` column on `classify_batches` stores raw exception text from `processBatchResults` failures; surfacing that verbatim to the manager UI would leak Anthropic response bodies / network errors. New `sanitiseBatchRow()` helper strips `notes` and replaces it with a boolean `has_processing_error` flag. Applied on every return path of GET.
- **`classify-batch` POST: tighter rate limit + 24h quota.** Per-minute cap dropped from 10 to 3 (a single submission already covers up to 1000 emails). Added a daily quota of 25,000 Anthropic batch requests per `submitted_by`, computed by summing `request_count` over the last 24h. Bounds the daily blast radius of a compromised manager session at a level well above any legitimate backfill.
- **`/api/tasks` PATCH and DELETE manager scope is now department-bound.** Read scope (`?scope=team`) was already department-scoped via the staff table; the write paths called `isManager()` with no department filter, so an Ops manager could PATCH/DELETE a Sales team task. Now both PATCH and DELETE check that the task's assignee or creator is in the manager's department before granting fall-through manager authority. super_admin remains unrestricted.
- **`staff-ai-prefs` PATCH logs misconfig** when a manager has `department = NULL` on their staff row (a likely cause of silent 403s after upgrade). The response stays a flat 403 to preserve the existing access-control shape; the log line lets ops diagnose it from Vercel function logs.
- **Cron-auth 500 response generic.** When `CRON_SECRET` is unset, the response body now says only `Internal server error`. The full diagnosis still goes to server logs, but the env-var name no longer appears in the HTTP response.
- **Stripped two leftover `any` casts** in `src/app/api/classify-email/route.ts` (lines 360, 520) on `feedback` and `senderHistory` map callbacks.

### Security

- JWT session signing: the `braiin_session` cookie is now a signed HS256 JWT instead of raw JSON. Forged cookies are rejected cryptographically at the proxy (edge) layer. Closes auth bypass where any non-empty cookie passed the previous presence check.
- Azure `access_token` and `refresh_token` removed from the JWT payload and the `Session` type. They were never read anywhere (all Graph calls use app-level `client_credentials`), so keeping them in the cookie only served as a leak vector for Mail.Read/Mail.Send scope tokens. If user-delegated Graph access is ever needed in future, store tokens in a dedicated encrypted table keyed by staff_id - not in the session cookie.
- `SESSION_SECRET` is now required at startup and must be at least 32 characters. The previous fallbacks (`SUPABASE_SERVICE_KEY`, literal `"fallback-dev-secret-change-me"`) have been removed - the app now refuses to start without a valid secret.
- Renamed `src/middleware.ts` to `src/proxy.ts` (Next 16 convention). The proxy cryptographically verifies the JWT via `verifySessionToken` on every `/api/*` request except `/api/auth/*` and `/api/cron/*`. Expired tokens return 401.
- Added shared `getSession()` helper in `src/lib/session.ts` that verifies the JWT and returns a typed `SessionPayload`. Replaced 7 duplicated route-local `getSession()` functions that did unsafe `JSON.parse` on the raw cookie value.
- `auth/session` endpoint now logs Supabase errors when looking up staff access rather than swallowing them silently.
- `send-email` now reads sender from the verified session, never from the request body. Closes the sender-spoofing vector where any authenticated user could send email appearing to come from any address. Also requires the session email to be an internal domain (403 otherwise), returns 401 if unauthenticated, and no longer leaks the full Resend error payload in the response.
- `email-composer` client no longer sends `from_email` / `from_name` (they were being ignored server-side after the fix).
- Consolidated Supabase client: 21 API routes now import `supabase` from `@/services/base` instead of each instantiating their own client. Closes the silent privilege-downgrade where `SUPABASE_SERVICE_KEY || NEXT_PUBLIC_SUPABASE_ANON_KEY` would fall through to anon if the service key was missing.
- `services/base.ts` now throws at startup if `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` are missing, or if `SUPABASE_SERVICE_KEY` is missing on the server. No more silent fallbacks.
