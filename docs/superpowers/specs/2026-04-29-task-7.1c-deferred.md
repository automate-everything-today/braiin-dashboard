# Task 7.1c deferred: event photos in draft system prompt

**Status:** Deferred. Phase 7 ships 7.1a (API) and 7.1b (events form upload) only.

## Why deferred

The current LLM gateway (`src/lib/llm-gateway`) accepts text-only `system` segments and a string `user` field. Anthropic's API supports image content blocks in user messages, but our gateway abstracts those away — `LlmMessage.content: string` and `user: string` provide no path to attach images.

Implementing 7.1c as planned would require:

1. Widening `LlmMessage.content` from `string` to `string | ContentBlock[]` (where `ContentBlock` covers text + base64 images).
2. Updating the anthropic provider in `src/lib/llm-gateway/providers/anthropic.ts` to translate the new shape.
3. Updating the content-hash cache in `src/lib/llm-gateway/cache.ts` so image bytes contribute to the hash deterministically.
4. Updating the metering in `src/lib/llm-gateway/metering.ts` so image-bearing calls are accounted correctly.
5. Adding tests across all four files.

That's a separate gateway-evolution epic — out of scope for the event follow-up data layer.

## What ships in Phase 7 anyway

- `event_media` table (migration 066) — populated by uploads.
- `/api/event-media` GET + POST (Task 7.1a).
- Photo strip + uploader on `/events` editor (Task 7.1b).
- Operator-memory-aid value (option b from the original brainstorm) — visible to operators reviewing contacts.

## What's missing

- AI-context value (option a from the original brainstorm) — Sonnet doesn't see the event photos when drafting follow-ups.

## When to revisit

Whenever the gateway is extended for multi-modal support (likely driven by a future feature elsewhere — e.g. document analysis, photo captioning for compliance). At that point, this task is small: read up to 3 photos for the event, base64-encode, attach to the user message in `generateDraft`.

The data is already captured. The pipeline is just paused at the LLM boundary.
